P0 modules are non-negotiable. If all six P0 modules are not live and working on Wednesday, the MVP is not done — regardless of how good everything else looks.

What to tell the tech team today — in this order
Open with this — the north star
The one sentence that frames everything
"MIP is not a dashboard tool. It is a marketing analyst. The AI output is the product. Every technical decision this week should ask: does this make the AI output faster, more accurate, or easier to act on? If it doesn't serve that — it's not in scope for MVP."
Clarify scope boundary
What is NOT in MVP scope
White-label reporting, PDF export, PPT export, 100+ connectors, predictive analytics, agentic actions (AI executing ad changes), A/B test tracking, audience overlap detection, LTV cohort analysis. None of these are in scope this week. If the team raises any of them — write it on the roadmap and move on.
AI system prompt — PM owns this
What you need to hand the dev team on Day 1
The system prompt draft. This is your job, not theirs. The dev team integrates the API — you write the instructions that go into it. The prompt must encode: (1) diagnostic tier order — ROAS first, then CTR/CVR, then creative, (2) India/SEA benchmarks by platform, (3) the 12 named root causes with trigger conditions and prescribed actions, (4) the health score formula, (5) the client paragraph format. Without this, the dev team cannot build a working AI feature.
Data model — agree on Day 1
The three data model decisions that block everything else
1. Brand profile schema — what fields does a client profile contain? (industry, objective, monthly spend, target CPA, platform mix). Agree this before any backend work starts.

2. Analysis output schema — what does the JSON response from the AI look like? (health_score, health_state, what_is_working, root_causes[], action_plan[], client_paragraph).

3. Workspace/client account hierarchy — follow PRD v1.1 exactly. No deviations without PM sign-off.
Blocker escalation rule
How blockers get raised and resolved
Any blocker that cannot be resolved within 2 hours gets escalated to you directly. You make the call — not the dev team. The rule: never let a blocker sit overnight. If the team is stuck on something on Friday at 5pm, that conversation must happen Friday, not Monday. A Monday discovery costs 60 hours.
Edge cases — answer before they ask
The five questions dev will hit mid-build
1. What happens when pasted data has no conversion columns? → Block analysis, show specific warning, list missing columns.

2. What if a client profile is incomplete? → Run analysis with a warning flag, note which context was missing.

3. What if ROAS data is missing but CTR/CVR data exists? → Run partial analysis, clearly label what was and wasn't assessed.

4. What if the AI returns a non-structured response? → Retry once, then show graceful error: "Analysis incomplete — please try again."

5. What platforms need to work at launch? → Meta Ads and Google Ads OAuth + manual CSV paste for everything else.
PM decisions you must make before dev starts — not during

These are the decisions that, if left open, the dev team will make using the wrong defaults. Make all six today.

Decision 1 — AI model
Which model powers the AI Brain?
Recommendation: Claude Sonnet (claude-sonnet-4-6). Best balance of analytical depth and response speed for the <30 second target. The wrong default is letting the dev team pick without PM input — they will optimise for cost, not quality. Quality of the AI output is the product. Own this decision.
Decision 2 — Output format contract
What does a valid MIP analysis output look like?
Define the exact JSON schema before build starts. Required fields: health_state (Healthy/At Risk/Critical), health_score (0–100), what_is_working (string), root_causes (array, min 1 max 3), action_plan (array, 3–5 items with reason + expected_impact), client_paragraph (string, 3–4 sentences). If the AI response doesn't contain all six — it is rejected and retried. This is a PM specification, not a dev decision.
Decision 3 — Input method priority
CSV paste only, or OAuth integrations too?
Both — but prioritise in this order. Manual CSV paste must work perfectly on Day 1. This is how Rohan uses MIP today and how the pilot demo runs. Meta Ads OAuth and Google Ads OAuth are Day 3–4 deliverables. If OAuth slips, paste still works and the demo is still possible. If you invert this priority and OAuth slips, you have nothing to show.
Decision 4 — Prototype fidelity
Does MVP match the prototype exactly?
No — and be explicit about this with the dev team. The prototype is the Phase 1 target interface. For MVP, the screens that must work are: Dashboard (health score overview), AI Brain (chat + analysis output), Integrations (connect Meta + Google + paste). Dashboards, Reports, Alerts, and full Settings can show "Coming soon" states. The demo path is: login → see client dashboard → open AI Brain → paste data → see output. That path must be flawless.
Decision 5 — Demo data
What data runs in the live demo?
Pick one real client. One real campaign export from the last 7 days. Rohan pastes it in the room — not a scripted demo, not a screen recording. Real data, live AI, in front of the agency owner. This means the system prompt must be tested on that exact client's data before Wednesday. Run the demo data through MIP at least 10 times during Days 5–6 QA and verify the output makes sense for that specific client context.
Decision 6 — Success definition
What does "MVP approved" look like?
The agency owner says yes to Phase 1 investment after seeing Rohan paste real data and receive a health score, root-cause diagnosis, action plan, and client paragraph in under 30 seconds — without any prompting or explanation from you. If you have to explain what the output means, the product hasn't done its job. That is your definition of done for Wednesday.
Definition of done — the exact gate criteria for Wednesday

MVP is done when Rohan can paste any Meta or Google export and receive all five outputs — health score, what's working, root cause with named diagnosis, action plan (3–5 numbered items), and client paragraph — in under 30 seconds. All five must be present. One missing = MVP is not done.

The 10-point launch checklist
AI output quality bar — what "good" looks like
Root cause specificity test
Wrong: "CTR is low."
Right: "CTR is 0.4% against India Meta benchmark of 0.8%. Frequency is 3.2 — this is creative fatigue. Recommended action: pause ad sets above frequency 3 and launch 2 fresh creative variants targeting the same audience. Expected CTR recovery: 7–10 days."
Client paragraph quality test
Wrong: "Your campaign performance needs improvement."
Right: "Your Meta campaigns delivered a ROAS of 2.8x last week, down from 3.6x the prior week, driven by creative fatigue on your top-performing ad set (frequency 3.4). We recommend pausing the fatigued creative and launching two fresh variants this week — expected to recover ROAS to 3.2x within 7 days."
Health score consistency test
Run the same data through MIP 3 times. The health score should be within ±3 points each time. If it varies by more than 5 points across runs on the same data — the prompt is non-deterministic and needs a temperature fix (set temperature to 0.2 or lower for scoring outputs).

If any of the 10 checklist items are incomplete on Wednesday morning, escalate immediately. A partial demo with acknowledged gaps is better than a failed live demo.
